# InvestLens 개발 컨벤션

이 문서는 InvestLens Backend의 변경, 검증, 커밋 기준을 정의합니다.

## 기본 원칙

- 기능 중심 모듈 구조와 `presentation/api -> application -> domain -> infrastructure` 의존 방향을 유지합니다.
- Controller는 Repository를 직접 호출하지 않습니다.
- 사용자 소유 데이터는 항상 사용자 ID가 포함된 쿼리로 접근합니다.
- 외부 RSS 및 AI 연동은 포트 인터페이스 뒤에 격리합니다.
- 새로운 의존성은 기존 코드와 JDK/Spring 기능으로 해결할 수 없는 경우에만 추가합니다.
- API 계약이나 동작을 변경할 때 관련 테스트와 문서를 함께 수정합니다.

## 변경 작업 절차

1. 변경 전 관련 코드와 테스트를 확인합니다.
2. 하나의 작업은 하나의 명확한 목적에 집중합니다.
3. 구현 후 변경 범위에 맞는 테스트를 먼저 실행합니다.
4. 최종적으로 `./gradlew clean build`를 실행합니다.
5. 검증이 통과한 변경만 커밋합니다.
6. 서로 다른 목적의 변경은 별도 커밋으로 분리합니다.

## 커밋 규칙

커밋 메시지는 기본적으로 한국어를 사용하고 다음 형식을 따릅니다.

```text
<type> : <명령형 요약>
```

허용하는 주요 type:

- `feat`: 사용자에게 보이는 기능 추가
- `fix`: 결함 수정
- `refactor`: 동작을 유지하는 구조 개선
- `test`: 테스트 추가 또는 수정
- `docs`: 문서 변경
- `chore`: 빌드, 설정, 도구, 운영 작업
- `perf`: 성능 개선
- `security`: 보안 강화

예시:

```text
feat : 포트폴리오 종목 등록 기능 추가
fix : 만료된 JWT 인증을 거부하도록 수정
docs : 로컬 실행 방법 보완
```

커밋에는 해당 기능의 테스트를 함께 포함합니다. 무관한 포맷 변경이나 문서 변경은 기능 커밋과 섞지 않습니다.

## 검증 기준

변경 범위에 따라 아래 순서로 검증합니다.

```bash
# 대상 테스트
./gradlew test --tests '<테스트 클래스>'

# 전체 검증
./gradlew clean build

# Git 공백 오류 확인
git diff --check
```

외부 시스템이 필요한 기능은 단위·통합 테스트 결과와 실제 PostgreSQL, RSS, Ollama 검증 여부를 구분해서 기록합니다.

## Git 운영

- 기본 원격 저장소는 `https://github.com/investlens-labs/InvestLens-BE.git`입니다.
- 작업 완료 시 에이전트는 별도 요청을 기다리지 않고 변경을 검토·검증한 뒤 목적별 커밋을 생성합니다.
- 원격 push는 테스트 통과 후 수행합니다.
- 강제 push, 기록 재작성, 기존 변경 삭제는 명시적인 요청 없이 수행하지 않습니다.
- 비밀키, `.env`, 운영 자격 증명, 로컬 데이터는 커밋하지 않습니다.

## 완료 보고

작업 완료 시 다음 내용을 남깁니다.

- 구현 또는 수정한 내용
- 실행한 검증과 결과
- 생성한 커밋 해시
- 원격 반영 여부
- 남아 있는 위험이나 검증하지 못한 외부 연동
